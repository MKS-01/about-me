a home server on 512 mb

written 2026-07-21 · 3 min

git clone github.com/MKS-01/pizow

I'm a seasonal hardware nerd. Sensors turn up, a weekend turns up, I build something and it goes in a drawer. This Pi Zero 2 W sat unused for months until one evening I flashed Ubuntu Server onto the card, mostly to find out whether a board with 512 MB of RAM could serve anything real or whether it was a toy.

It turned into the server my house runs on. Here's what's actually on it.

what it is

pizow is a set of shell scripts and one Next.js dashboard. setup-pi.sh takes a blank Ubuntu install to a working server — Node 22, PM2, nginx, and a swap file — and it's idempotent, so re-running it after you've broken something is the repair path.

After that deploy.sh is the only thing I run, and it has three modes because the Pi can't always do the work itself:

$ deploy.sh --local     # build on the mac, rsync the result
$ deploy.sh --remote    # git pull + build on the pi
$ deploy.sh --restart   # just bounce pm2

nginx fronts the app on port 80 and PM2 keeps it alive across reboots. The dashboard reports CPU, memory, disk, temperature, network and every running service, polling its own API rather than holding a socket open — on a board this size, an idle connection per browser tab is a real cost.

what 512 mb forces

Most of what this project taught me was the hardware refusing to be ignored.

  • Swap isn't optional. Builds fail outright without it, so setup-pi.sh creates a 1 GB swapfile before anything else. 512 MB is not enough to compile a Next.js app, and the failure mode is the OOM killer taking the build out mid-run rather than a clean error.
  • Build somewhere else when you can. Next.js output: 'standalone' emits a self-contained bundle, so --local rsyncs .next/standalone and the Pi runs node server.js directly. No npm start, no node_modules install on the device, no toolchain. The Pi only executes.
  • The USB port caps at 500 mA. The OTG port will not spin a mechanical drive. Flash storage or a powered hub — this one cost me an evening before I read the spec.
  • Assume the disk is missing. The NAS mounts at /mnt/nas with nofail, so the Pi still boots when the drive isn't attached. Anything reading from it checks mountpoint -q first instead of trusting the path.

the nas half

setup-nas.sh turns any USB stick into shared storage: format to ext4, export over NFS for machines that speak it, and run File Browser on port 8080 for the ones that don't. A udev rule remounts the drive when it's pulled and reinserted, which matters more than it sounds — without it, one accidental unplug leaves you SSH'ing in to remount by hand.

It runs independently of the app. Nothing in the web stack depends on the drive being present, which is the same nofail instinct applied at the architecture level.

it serves readback

The Pi is where readback's library lives. That split works because of how readback is built: audio generation wants a GPU and stays on the Mac, but the library dashboard is model-free — it lists rows out of SQLite and serves a finished WAV.

That's a static-file workload with a database index in front of it, which is exactly what a 512 MB board is good at. The expensive machine does the expensive thing; the cheap machine does the thing that runs all day. Neither is doing work it's badly suited for.

where it landed

The commit log is honest about the rest: a burst of work, then nothing for months, then a second burst when I had something real to point at it. I don't maintain things I'm not using, and the Pi kept running the whole time — it just didn't need me. It's the oldest thing in my weekend-hacks folder and the one I use most, and it started as a board I had no plan for.
